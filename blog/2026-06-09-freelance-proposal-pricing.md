# 技术自由职业者如何报价：把模糊需求变成可成交提案

> 目标平台：掘金 / 知乎 / V2EX / 公众号  
> 标签：自由职业、AI 自动化、报价、接单、技术服务  
> 配套项目：Proposal Generator

---

很多技术人接自由职业单子时，最难的不是写代码，而是报价。

客户往往不会这样说：

> 我需要一个 FastAPI + PostgreSQL + Celery + Playwright 的竞品价格监控系统，支持定时任务、异常重试、代理池、微信通知和部署文档。

客户更常见的说法是：

> 能不能帮我做个自动看竞品价格的东西？每天提醒我一下。

这句话背后可能是一个 ¥800 的脚本，也可能是一个 ¥30,000 的监控系统。区别不在技术栈，而在范围：目标网站数量、是否登录、是否反爬、是否要数据库、是否要告警、是否多人使用、是否长期维护。

所以自由职业者必须掌握一个核心能力：**把模糊需求翻译成可报价的交付范围。**

---

## 一、为什么技术服务报价经常失控？

### 1. 客户说的是目标，不是需求

客户说“自动抓取价格”，真实需求可能包含：

- 从哪些网站抓？
- 抓哪些字段？
- 多久抓一次？
- 是否需要登录？
- 是否有验证码？
- 数据保存在哪里？
- 怎么提醒？
- 出错谁处理？

如果不问清楚，报价一定会失真。

### 2. 技术人容易按“代码量”报价

很多人会想：“这个脚本我一天就能写完，收 ¥500 吧。”

但客户买的不是代码行数，而是业务结果：

- 节省人工盯价格的时间
- 比竞争对手更快响应促销
- 形成长期可追踪的数据资产
- 减少漏看、误看、晚看的风险

如果只按代码量报价，你会把自己的价值压到最低。

### 3. 没有边界，就没有利润

最常见的亏本单是这样开始的：

> 很简单，帮我做个小工具。

然后逐步变成：

- 顺便加个后台
- 顺便支持多个账号
- 顺便部署到服务器
- 顺便做个微信通知
- 顺便帮我维护一下

每个“顺便”都是范围扩张。如果没有提前定义交付边界，最后就是加班买单。

## 二、一个简单但有效的报价框架

我建议把所有技术服务都拆成三档：基础版、标准版、高级版。

### 基础版：验证需求

适合客户预算有限、需求不复杂的情况。

特点：

- 单一流程
- 少量数据源
- 手动或半自动运行
- 简单文档
- 不承诺复杂异常处理

示例报价：

- 自动化脚本：¥800 - ¥2,000
- 数据采集：¥1,500 - ¥3,000
- AI Bot Demo：¥3,000 - ¥8,000
- RAG 知识库 Demo：¥5,000 - ¥10,000

基础版的目标不是做“大而全”，而是快速验证客户是否真的需要这个东西。

### 标准版：可稳定使用

这是最推荐的成交档位。

特点：

- 有定时任务
- 有日志和异常处理
- 有数据库或结构化存储
- 有部署文档
- 有基础告警
- 可交给客户日常使用

示例报价：

- 竞品监控系统：¥5,000 - ¥12,000
- AI Agent 工具调用：¥12,000 - ¥30,000
- 企业知识库：¥15,000 - ¥30,000
- MCP 系统集成：¥15,000 - ¥30,000

标准版卖的是“可靠性”。这也是客户真正愿意付费的地方。

### 高级版：业务系统化

适合企业客户或长期项目。

特点：

- Web 管理后台
- 多用户/权限
- 监控告警
- 审计日志
- 私有化部署
- 长期维护

示例报价：

- 企业 Agent 平台：¥50,000+
- 竞品情报平台：¥30,000+
- 企业 MCP 网关：¥50,000+
- 私有化知识库：¥50,000+

高级版不要轻易低价接，因为它本质上已经不是“脚本”，而是“系统”。

## 三、报价前必须问的 5 类问题

### 1. 数据来源

- 数据来自网页、Excel、API 还是数据库？
- 是否需要登录？
- 是否有验证码？
- 数据量大概多少？

这决定技术复杂度。

### 2. 运行频率

- 手动运行？
- 每天一次？
- 每小时一次？
- 实时触发？

频率越高，对稳定性、日志、异常恢复要求越高。

### 3. 交付形态

- 命令行脚本？
- Web 页面？
- API 服务？
- Docker 部署？
- 云服务器上线？

交付形态直接决定报价档位。

### 4. 权限与安全

尤其是 AI Agent / MCP / 企业知识库项目，必须问：

- 是否访问内部系统？
- 是否涉及客户数据？
- 是否允许写操作？
- 是否需要审计日志？

如果 Agent 能“查数据”还好；如果 Agent 能“改数据/删数据/发消息”，安全成本就完全不同。

### 5. 维护方式

- 交付后是否需要维护？
- 第三方网站变化谁处理？
- API 费用谁承担？
- 服务器费用谁承担？

维护必须单独收费。否则一次性项目会变成永久客服。

## 四、把报价写成客户能理解的提案

一个好提案不应该只有价格，而应该包含：

1. 需求理解
2. 推荐方案
3. 三档报价
4. 交付物
5. 需要客户确认的问题
6. 风险与边界
7. 下一步行动

比如客户说：

> 帮我做一个自动抓取竞品价格并每天发微信提醒的工具。

你可以这样回复：

> 你好，我看了你的需求，初步判断这是一个「数据采集 / 竞品监控」项目。建议先做标准版，预算大概在 ¥5,000 - ¥12,000，周期约 1-2 周。交付内容包括多站点采集、定时任务、数据库存储、降价/变化提醒。开工前我需要确认：目标网站 URL、采集字段、每天数据量、是否需要登录或代理。如果范围确认，我可以给你出详细执行方案和里程碑。

这段话比“可以，¥2000”专业很多，也更容易筛选出真正有预算的客户。

## 五、我做了一个小工具：Proposal Generator

为了让报价流程更标准化，我做了一个零依赖 Python 工具：Proposal Generator。

它可以根据一句客户需求，自动输出：

- 项目类型判断
- 推荐报价档位
- 三档价格范围
- 交付物清单
- 开工前问题
- 风险边界
- 可直接发给客户的话术

使用方式：

```bash
python proposal_generator.py "帮我做一个自动抓取竞品价格并每天发微信提醒的工具"
```

输出会包含：

```text
项目类型：数据采集 / 竞品监控
推荐档位：标准版
推荐报价：¥5,000 - ¥12,000
预计周期：1-2 周
```

再比如：

```bash
python proposal_generator.py "把 CRM 系统封装成 MCP 工具，让 Claude 可以查客户数据"
```

它会识别为 MCP 协议集成，并生成对应的报价和安全问题。

## 七、最后的建议

技术自由职业者想赚到钱，不能只会写代码，还要会定义范围、表达价值、控制风险。

记住三句话：

1. **不要给模糊需求直接报价。** 先问问题。
2. **不要只卖代码。** 要卖业务结果和可靠性。
3. **不要没有边界地交付。** 范围之外就是新需求。

报价不是拍脑袋，而是一套工程化流程。

当你能把客户一句模糊的话，变成一份清晰的提案时，成交概率和项目利润都会明显提高。
