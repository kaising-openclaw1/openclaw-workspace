> 版本：v0.2（企划版）· 作者：小鸣
> 目标：构建多机器 Agent 算力操作系统，对标 Claude Code 架构层级，超越其单机限制

## 一、核心理念

### 1.1 一句话定义

**Agent OS 是一个分布式操作系统，将多台机器的算力（CPU/GPU/模型API）抽象为统一资源池，让 Agent 在任意节点上以最高智力运行，同时保证源码安全。**

### 1.2 与 Claude Code 的根本区别

| 维度 | Claude Code | Agent OS |
|------|------------|----------|
| 架构 | 单进程 CLI 工具 | **分布式操作系统** |
| 算力 | 单机 + 单模型 API | **多机算力池 + 多模型分级** |
| 安全 | 无内置加密 | **安全飞地 + 审计追踪** |
| 扩展 | 内置工具集 | **插件系统 + 热加载** |
| 协作 | 单人单机 | **多节点 Mesh 协作** |
| 状态 | 无状态（每次重读） | **状态机 + Checkpoint/Restore** |

### 1.3 设计原则

```
1. 算力即资源    → 所有计算能力统一抽象，按需分配
2. 智力最大化    → 简单任务不浪费强模型，复杂任务不妥协
3. 安全不可妥协  → 代码加密存储、运行时解密、全审计
4. 去中心化优先  → Mesh 架构，任何节点可做控制面
5. 可观测内建    → 从第一天起全链路追踪
6. 工程卓越      → 类型安全、全面测试、优雅 API
```

## 三、核心子系统详细设计

### 3.1 智能路由层（Intelligence Router）

这是整个系统的"大脑"，决定每个任务用哪个模型。

#### 3.1.1 任务画像引擎

```
输入：任务描述 / 代码 / 问题
  │
  ├── 特征提取
  │   ├── Token 计数（输入/输出预估）
  │   ├── 复杂度分析（关键词匹配 + ML 分类）
  │   ├── 工具需求检测（是否需要调用外部工具）
  │   ├── 多模态检测（是否需要看图/音频）
  │   └── 安全等级评估（是否涉及敏感数据）
  │
  ├── 约束收集
  │   ├── 最大成本预算（用户设定）
  │   ├── 最大延迟容忍（用户设定）
  │   ├── 最低智力要求（用户设定）
  │   └── 合规要求（数据不出境等）
  │
  └── 输出：TaskProfile
```

#### 3.1.2 智力等级定义（5 级）

| 等级 | 名称 | 典型模型 | 适用场景 | 成本/1k tokens |
|------|------|---------|---------|---------------|
| 0 | TINY | 规则引擎 / 正则 | 关键词提取、格式化 | $0.00001 |
| 1 | LIGHT | Qwen2.5-7B / Gemma-2B | 基础问答、简单分类 | $0.00005 |
| 2 | MEDIUM | DeepSeek-V3 / GPT-4o-mini | 一般推理、代码补全 | $0.0003 |
| 3 | HIGH | GPT-4o / Claude Sonnet | 复杂推理、代码生成 | $0.005 |
| 4 | MAXIMUM | Claude Opus / o3 / DeepSeek-R1 | 深度研究、架构设计 | $0.015 |

#### 3.1.3 多目标优化算法

```
评分函数：Score(model) = w1 * Capability + w2 * (1 - Cost/CostMax) + w3 * (1 - Latency/LatencyMax) + w4 * Reliability

权重默认值：
  - Capability: 0.40（能力最重要）
  - Cost:       0.25（成本第二）
  - Latency:    0.20（延迟第三）
  - Reliability:0.15（可靠性保底）

动态调整：
  - 用户明确要求"省钱" → w2 提升到 0.40
  - 用户明确要求"最快" → w3 提升到 0.40
  - 历史失败率 > 20% → 自动降级到次优模型
```

#### 3.1.4 降级策略

```
Level 4 (MAX) → Level 3 (HIGH) → Level 2 (MEDIUM) → Level 1 (LIGHT) → Level 0 (TINY)

降级触发条件：
1. 目标等级无可用模型
2. 成本超预算
3. 连续 3 次失败
4. API 限流/超时

升级触发条件：
1. 任务在低等级连续失败 2 次（复杂度被低估）
2. 用户明确要求"用最好的"
```

### 3.2 Mesh 网络层

#### 3.2.1 节点角色

```
┌──────────────────────────────────────────────────┐
│                   Mesh 集群                        │
│                                                    │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐        │
│  │Controller│───│  Worker  │───│  Worker  │        │
│  │ (调度)   │   │ (执行)   │   │ (执行)   │        │
│  └────┬────┘   └─────────┘   └─────────┘        │
│       │                                            │
│  ┌────┴────┐   ┌─────────┐   ┌─────────┐        │
│  │ Gateway │───│  Edge   │───│  Relay  │        │
│  │ (外网)  │   │ (受限)  │   │ (中转)  │        │
│  └─────────┘   └─────────┘   └─────────┘        │
└──────────────────────────────────────────────────┘
```

| 角色 | 职责 | 典型配置 |
|------|------|---------|
| **Controller** | 任务调度、资源分配、状态管理 | 高性能服务器 |
| **Worker** | 执行 Agent 任务、运行沙箱 | 有 GPU 的机器 |
| **Hybrid** | 同时承担 Controller + Worker | 主力开发机 |
| **Gateway** | 对外暴露 API、NAT 穿透 | 有公网 IP 的机器 |
| **Edge** | 轻量任务、数据采集 | 树莓派/手机 |
| **Relay** | 中转流量、协议转换 | 云服务器 |

#### 3.2.2 发现协议

```
阶段 1：局域网发现（mDNS/Broadcast）
  Node A 广播 → "我是 node-a，端口 8765"
  Node B 收到 → "我在！建立连接"
  → 直接 TCP 连接

阶段 2：种子节点（公网）
  Node A → 连接种子节点 S
  S 返回已知节点列表 [B, C, D]
  A → 尝试连接 B, C, D

阶段 3：NAT 穿透
  如果直接连接失败 → 尝试 STUN
  如果 STUN 失败 → 通过 Relay 中转
  如果 Relay 也失败 → Gateway 代理

阶段 4：Gossip 同步
  每 30s 广播心跳 + 已知节点列表
  新节点在 2 轮 gossip 内全网可见
```

#### 3.2.3 通信协议

```
传输层：gRPC (双向流) + WebSocket (浏览器)
序列化：Protocol Buffers
加密：mTLS (生产) / Noise Protocol (轻量)
压缩：Snappy / Zstandard

消息类型：
  - Control: Join/Leave/Heartbeat/Election
  - Task: Submit/Cancel/Status/Output
  - Data: FileTransfer/ResourceReport
  - Sync: StateSync/LogReplication
```

### 3.3 安全飞地（Security Enclave）

#### 3.3.1 加密架构

```
┌─────────────────────────────────────────────┐
│              安全飞地                          │
│                                               │
│  主密钥 (Master Key)                          │
│    ├── HKDF(context="source_code")           │
│    │   └── 文件密钥 1 → AES-256-GCM 加密文件1│
│    │   └── 文件密钥 2 → AES-256-GCM 加密文件2│
│    ├── HKDF(context="config")                │
│    │   └── 配置密钥 → 加密配置文件            │
│    └── HKDF(context="communication")         │
│        └── 会话密钥 → 加密 Mesh 通信          │
│                                               │
│  密钥缓存：LRU，1h TTL                        │
│  密钥轮换：每 30 天自动轮换主密钥              │
└─────────────────────────────────────────────┘
```

#### 3.3.2 代码保护流程

```
开发阶段：
  源码 → 加密存储 → .enc 文件（可提交到 Git）
  只有授权 Agent 可解密运行

运行阶段：
  Agent 请求执行 → 飞地验证权限
  → 解密到内存（不写磁盘）
  → Agent 在沙箱中执行
  → 沙箱销毁，内存清零

审计追踪：
  每次解密 → 记录 (who, what, when, result)
  每次执行 → 记录 (agent_id, file_hash, exit_code)
  日志不可篡改（HMAC 链式校验）
```

#### 3.3.3 RBAC 模型

```
主体 (Subject): User / Agent / Service
资源 (Resource): file:path / api:endpoint / compute:gpu
动作 (Action): read / write / execute / admin
条件 (Condition): time_range / ip_range / mfa_required

策略示例：
  {
    "subject": "agent:code-reviewer",
    "resource": "repo:my-project/**",
    "action": "read",
    "condition": {"time": "09:00-18:00"}
  }
```

### 3.4 编排引擎（Orchestration Engine）

#### 3.4.1 事件总线设计

```
┌─────────────────────────────────────────────┐
│              事件总线                          │
│                                               │
│  发布者 (Publisher)                           │
│    ├── 系统事件 (system.*)                    │
│    ├── 任务事件 (task.*)                      │
│    ├── 节点事件 (node.*)                      │
│    └── 安全事件 (security.*)                  │
│                                               │
│  队列 (4 级优先级)                            │
│    ├── CRITICAL (0) → 立即处理                │
│    ├── HIGH (1)    → <100ms                   │
│    ├── NORMAL (2)  → <1s                      │
│    └── LOW (3)     → 可延迟                   │
│                                               │
│  订阅者 (Subscriber)                          │
│    ├── 精确匹配 ("task.completed")            │
│    ├── 通配符 ("task.*")                      │
│    └── 条件过滤 (filter_fn)                   │
│                                               │
│  背压控制                                     │
│    ├── 队列满 → 丢弃 LOW 事件                 │
│    ├── 持续满 → 触发告警                      │
│    └── 死信队列 → 持久化未处理事件             │
└─────────────────────────────────────────────┘
```

#### 3.4.2 状态机引擎

```
状态定义（11 种）：
  PENDING → INITIALIZING → RUNNING ⇄ PAUSED
  RUNNING → WAITING → RUNNING
  RUNNING → COMPLETED / FAILED / TIMEOUT
  FAILED → ROLLING_BACK → ROLLED_BACK

转换矩阵（严格校验）：
  State[from] → Set[allowed_to_states]
  非法转换 → TransitionError

Checkpoint 机制：
  每 N 步自动保存状态快照
  支持从任意 checkpoint 恢复
  用于长时间任务的中断恢复
```

#### 3.4.3 任务调度器

```
调度策略：
  1. 优先级队列（0-9，0 最高）
  2. 资源感知调度（CPU/GPU/内存约束）
  3. 亲和性调度（同一项目任务尽量在同一节点）
  4. 负载均衡（避免单节点过载）

调度算法：
  Score(node) = w1 * (1 - Load) + w2 * Affinity + w3 * DataLocality
  
  其中：
  - Load: 节点当前负载 (0-1)
  - Affinity: 历史亲和度 (0-1)
  - DataLocality: 数据本地性 (0-1)
```

#### 3.6.2 沙箱执行环境

```
三种沙箱模式：

1. Python Sandbox（轻量）
   - subprocess + 资源限制（cgroups）
   - 适合：脚本执行、数据处理
   - 隔离级别：进程级

2. Docker Sandbox（标准）
   - 每个任务启动临时容器
   - 适合：需要依赖安装的任务
   - 隔离级别：容器级

3. WASM Sandbox（安全）
   - WebAssembly 运行时
   - 适合：不受信代码执行
   - 隔离级别：沙箱级

选择策略：
  - 用户代码 → Docker Sandbox
  - 系统工具 → Python Sandbox
  - 第三方插件 → WASM Sandbox
```

## 六、关键技术决策

### 6.1 为什么选 Python？

| 维度 | Python | Rust/Go | 决策 |
|------|--------|---------|------|
| 开发速度 | ✅ 快 | ❌ 慢 | Python |
| AI/ML 生态 | ✅ 最好 | ❌ 差 | Python |
| 并发 | ❌ GIL | ✅ 原生 | 用 asyncio 弥补 |
| 性能 | ❌ 慢 | ✅ 快 | 核心用 Rust 扩展 |
| 学习成本 | ✅ 低 | ❌ 高 | Python |

**结论：Python 为主，核心性能热点用 Rust 写扩展（类似 Pydantic v2 策略）。**

### 6.2 为什么 Mesh 而不是 Master-Slave？

| 维度 | Mesh | Master-Slave |
|------|------|-------------|
| 单点故障 | ❌ 无 | ✅ 有 |
| 扩展性 | ✅ 线性 | ❌ 有限 |
| 复杂度 | ❌ 高 | ✅ 低 |
| 运维 | ❌ 难 | ✅ 易 |

**结论：Mesh 更适合我们的场景——开发者的机器随时可能关机/休眠，不能依赖固定主节点。**

### 6.3 为什么安全飞地而不是全盘加密？

| 维度 | 安全飞地 | 全盘加密 |
|------|---------|---------|
| 粒度 | ✅ 文件级 | ❌ 磁盘级 |
| 性能 | ✅ 按需解密 | ❌ 全量解密 |
| 审计 | ✅ 完整 | ❌ 无 |
| 密钥管理 | ✅ 灵活 | ❌ 单一 |

**结论：安全飞地更适合保护源码——不同项目用不同密钥，细粒度控制谁可以看什么。**

## 十、总结

**Agent OS 不是一个"更好的 Claude Code"，而是一个全新的品类：分布式 Agent 算力操作系统。**

Claude Code 解决的是"单个开发者如何用 AI 写代码"，Agent OS 解决的是"如何用多台机器的算力安全地运行多个 AI Agent"。

这是两个不同层次的问题。Claude Code 是工具，Agent OS 是基础设施。
